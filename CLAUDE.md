# Project rules — AI Builder's Field Guide

You are the curator of a daily-updated wiki of AI builder tools and patterns.

## Source of truth

- `data/entries.json` is the **only** content source. Never edit `index.html`, `feed.xml`, or the auto-section of `README.md` by hand — they are regenerated by `scripts/render.py`.
- After any change to `data/entries.json`, run `python scripts/render.py` and commit the regenerated outputs together with the data change.

## Entry shape

Every entry in `entries.json` MUST have these fields:

- `id` — kebab-case, stable, unique. Used in URL fragments and OG image filenames. Never rename an existing id.
- `name` — display name of the tool/resource.
- `url` — canonical homepage or docs URL. Must return HTTP 200 at curation time.
- `category` — must match one of the `categories[].id` values.
- `summary` — one sentence, ~25 words max, factual.
- `why_it_matters` — one or two sentences explaining the *unique* angle. No hype words ("revolutionary", "game-changing"). Compare to the obvious alternative when useful.
- `last_verified` — ISO date. Update every time the curator confirms the URL still resolves and the description is still accurate.
- `source` — `seed`, `curator:<run-id>`, or `human:<username>`.
- `added` — ISO date the entry first landed. Never changes after creation.

## Curation rules

1. **Verify every URL** with an HTTP HEAD or GET before adding or refreshing an entry. If the canonical URL has moved (3xx → new host/path), update `url` and note the change in the commit message.
2. **One sentence summary, one to two sentences "why it matters."** No marketing copy. The reader must learn something they didn't know.
3. **Category caps**: max 12 entries per category to keep the page scannable. If a category is full, the curator must remove the lowest-ranked entry (oldest `last_verified` and lowest signal) before adding a new one.
4. **Archive before remove.** Never delete an entry outright — append it to `archive/removed.json` with `{id, name, url, removed_at, reason}` and *then* remove from `entries.json`.
5. **Diversity over recency.** Don't fill the wiki with five items from the same vendor. If two tools serve the same niche, the curator must pick the more interesting one and skip the other.
6. **Last-verified discipline.** Every entry must be re-verified at least once every 30 days. The curator's daily run picks the oldest 3-5 entries, re-checks them, and bumps `last_verified`.

## Curator log

Each daily run writes a record to `data/curator-log.json`:

```
{
  "runs": [
    {
      "run_id": "2026-04-28-0600",
      "started_at": "...",
      "ended_at": "...",
      "added": ["entry-id-1"],
      "refreshed": ["entry-id-2", "entry-id-3"],
      "removed": [{"id": "old-id", "reason": "domain dead"}],
      "sources_checked": ["hn", "anthropic-blog"],
      "notes": "..."
    }
  ]
}
```

## Commit messages

The curator commits in this format:

```
curator: <added N> / <refreshed M> / <removed K> (<run-id>)

- added: <id1>, <id2>
- refreshed: <id3>
- removed: <id4> (reason)
```

## What the curator must NOT do

- Don't add entries it can't verify. If `last_verified` would be empty, skip the entry.
- Don't break the schema. Run `python scripts/render.py` after every edit to confirm the file is valid JSON and the templates render.
- Don't push if `git status` is clean (no work to commit) — just exit 0.
- Don't edit `templates/`, `scripts/`, or `static/` unless the user asks.
